- **[EN]** Embedding indexing no longer breaks on the cloud path. Added a Firestore rule for `users/{uid}/embeddings/{docId}` (owner read/write) — without it writes/reads returned `Missing or insufficient permissions` and broke the background run. The indexer's freshness check is now local-only (no cloud read in the loop, where it could throw permissions or `LOCKED` when E2E is locked). Cloud read/write of embeddings is best-effort: failures no longer break local indexing, and `ENCRYPT_REQUIRED` (E2E locked) is logged quietly — the embedding is stored locally and syncs later. Rules deployed to both databases.

## 2026-06-12 (v0.7.18)
- **[RU]** Исправлено нарушение CSP `script-src` при открытии заметки из архива и нажатии «Продолжить писать». Чанк назывался `vendor-ai`, но AI SDK не при чём — внутри `zod` v4 (4.3.6): его JIT-парсер пробует `new Function("")` для feature-detection, что наша CSP (без `unsafe-eval`) блокирует и логирует как ошибку. Всплывало именно на этом пути, т.к. там первый JIT-пригодный `safeParse` (`VersionService.getLatestVersion` → `versionDbSchema`). Отключил JIT через `z.config({ jitless: true })` в `main.tsx` — короткое замыкание не доходит до `new Function`, парсинг идёт интерпретатором, CSP остаётся жёсткой. Гейт: `tsc` 0, прод-сборка зелёная.
- **[EN]** Fixed a CSP `script-src` violation when opening a note from the archive and clicking "Continue writing". The chunk was named `vendor-ai`, but the AI SDK is unrelated — it actually contains `zod` v4 (4.3.6): its JIT parser probes `new Function("")` for feature detection, which our CSP (no `unsafe-eval`) blocks and logs as an error. It surfaced on this path because it holds the first JIT-eligible `safeParse` (`VersionService.getLatestVersion` → `versionDbSchema`). Disabled the JIT via `z.config({ jitless: true })` in `main.tsx` — the short-circuit never reaches `new Function`, parsing uses the interpreter, and the CSP stays hardened. Gate: `tsc` 0, production build green.
- **[RU]** «Продолжить писать» из архива снова работает. Раньше всплывал тост «Ошибка при продолжении сессии»: `handleContinueSessionOrDoc` обрабатывал только `LifeLogDocument` (путь лайфлога), а архив передаёт `ArchiveSession` — попадало в `else` и сразу слало ошибку, ветку для архива не доделали. Теперь `ArchiveSession` конвертируется в `LifeLogDocument` (флаги хранения `_isLocal`/`_linkedCloudId` → `localId`/`cloudId`) и проходит через тот же `handleContinueDocument`: локальная заметка грузится из локального хранилища, облачная подтягивает локальную копию. Чинит и десктоп, и мобайл-архив.
- **[EN]** "Continue writing" from the archive works again. It used to show an "Error continuing session" toast: `handleContinueSessionOrDoc` only handled `LifeLogDocument` (the LifeLog path), while the archive passes an `ArchiveSession` — it fell into the `else` and raised an error, as the archive branch was never finished. Now the `ArchiveSession` is mapped to a `LifeLogDocument` (storage flags `_isLocal`/`_linkedCloudId` → `localId`/`cloudId`) and goes through the same `handleContinueDocument`: a local note loads from local storage, a cloud note pulls a local copy.
