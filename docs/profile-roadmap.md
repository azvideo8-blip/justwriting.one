# Живой AI-профиль — роадмап (для исполнителя / opencode)

Документ самодостаточный: контекст + что уже сделано + что осталось, тикетами. Цель — довести «темы профиля» (фасеты) до хорошего качества и связать их с чатом / постами.

Проект: `justwriting` (React + TS + Vite; бэкенд — Firebase Functions; деплой — Vercel `just-write-test` → justwriting.vercel.app). Провайдер ИИ — **Fireworks** (`AI_PROVIDER=fireworks`), активная чат-модель `deepseek-v4-flash`.

---

## 0. Общий контекст (читать перед всем)

**Что такое фасеты:** профиль пользователя строится как набор тем («фасетов»). Сейчас две группы: (а) заданные **жизненные домены** (Деньги, Отношения с детьми/партнёром/родителями/коллегами, Самореализация), (б) эмерджентные «прочие» темы из кластеризации остатка.

**Конвейер (всё уже есть):**
1. Заметки индексируются по **чанкам**: `embedDocument` (Fireworks `qwen3-embedding-8b`, 1024 dim) режет заметку на куски (~1000 симв, overlap 150), возвращает `{ vectors, chunks(тексты), model, dim }`. Хранится в `aiEmbeddings` (IndexedDB, схема БД v7; embedding schemaV=3 — векторы + тексты чанков). Облачная копия зашифрована.
2. Кластеризация/назначение — **клиентская** (E2E: векторы локальные). `AIProfileFacetService.build()`: для каждого **чанка** argmax-назначение в ближайший домен (cosine ≥ `DOMAIN_THRESHOLD`); остаток → k-means «прочие». Каждый фасет описывается ИИ по СВОИМ кускам.
3. Саммари фасета — `summarizeFacet` (Cloud Function), модель `AI_FACET_MODEL` (по умолч. `qwen3-30b-a3b`), формат `НАЗВАНИЕ:/ОПИСАНИЕ:`, фокус по теме.
4. UI — `ProfileFacets.tsx` в Диагностике → «Профиль AI»: карточки (название, описание, кол-во заметок, диапазон дат, бейдж тренда), кнопки «Построить/Перестроить» и «Выгрузить».

**Ключевые файлы:**
- `src/features/ai/services/AIProfileFacetService.ts` — оркестрация build (назначение, кластеры, саммари, стор).
- `src/features/ai/utils/facetClustering.ts` — `clusterChunks`, `mergeSimilarClusters`, `suggestK`, `cosineSimilarity` (последняя — в `vectorSearch.ts`).
- `src/features/ai/utils/lifeDomains.ts` — `LIFE_DOMAINS` (id, label, seed-текст).
- `src/features/ai/components/ProfileFacets.tsx` — UI карточек.
- `functions/src/ai/summarizeFacet.ts` — LLM-саммари (FACET_MODEL, focus, парс/реджект рассуждений).
- `src/core/storage/localDb.ts` — `AIProfileFacet`, стор `aiProfileFacets` (БД v7); `AIDocumentEmbedding` (+`chunkTexts`, schemaV 3).
- `src/features/ai/utils/embeddingIndexer.ts` — `CURRENT_EMBED_SCHEMA=3`, фоновый индексатор (`useEmbeddingIndexer.ts`).
- `src/features/ai/services/AIService.ts` — `embed`, `summarizeFacet`, `rerank`, `chat`.
- `src/features/ai/hooks/useAIChat.ts` — чат + «липкий» поиск по заметкам (точка для интеграции фасетов).

**Тюнинг-константы** (в `AIProfileFacetService.ts`): `DOMAIN_THRESHOLD=0.45`, `MERGE_THRESHOLD=0.80`, `MAX_DISCOVERED=6`, `MIN_FACET_NOTES=2`, `MAX_EXCERPTS=12`, `EXCERPT_CHARS=1200`. В `facetClustering.ts`: `suggestK` = clamp(notes/4, 4, 18).

**ГЛАВНЫЕ ГРАБЛИ (не наступать):**
- **Инфраструктурные ИИ-вызовы** (`embedDocument`, `rerankNotes`, `summarizeFacet`) НЕ должны вешать `checkDailyLimit` (5/сутки) и `checkRateLimit` (кулдаун 10с) — только `withinGlobalDailyLimit()` + `recordUsage`. Иначе `429` и блок массовых прогонов. Эталон — эти три функции.
- **DeepSeek v4 — reasoning-модель**: сыплет рассуждения в `content` («Мы должны проанализировать… похоже…»). Для структурных задач (фасеты) использовать `AI_FACET_MODEL` (не активную чат-модель) + жёстко отклонять рассуждения в парсере. Эталон — `summarizeFacet.ts` (META-regex, срез преамбулы после «ОПИСАНИЕ:»).
- **Деплой функций:** `cd functions && firebase deploy --only functions:<name>`. Новая v2-callable требует `allUsers`→`run.invoker` (Firebase ставит сам; проверка: `curl -X POST <url> -d '{"data":{}}'` → 401 = достижима).
- Не нарушать слоевые границы (eslint `import/no-restricted-paths`). Гейт перед коммитом: `npx tsc --noEmit` (root + `functions`), `npx eslint <файлы>`, `npm run build`.
- Фасеты сейчас **локальные** (регенерируемы), облачного синка нет.

## Текущее состояние

**СДЕЛАНО:** chunk-level эмбеддинги; назначение чанков в домены; саммари фасетов послушной моделью; чистый keyword-фолбэк; отсев шумных «прочих»; UI карточек (раскрытие, выгрузка, тренды); стор `aiProfileFacets`.

**НЕ РЕШЕНО (отсюда роадмап):**
1. Качество саммари только что переведено на `qwen3-30b-a3b` — **нужна проверка на реальных данных** (см. PROF-1).
2. Домены **«Дети/Родители/Коллеги» не появляются** — назначение argmax + широкие сиды отдают их куски в «Самореализацию» (PROF-2).
3. Нет живого обновления, связки с чатом, кандидатов в посты, лиц людей (PROF-4..7).

---

## PROF-1 — Проверка качества саммари + тюнинг порогов
**Цель:** убедиться, что после перехода на `qwen3-30b-a3b` карточки получают чистые осмысленные описания (без «мы должны проанализировать», без keyword-мусора).
**Действия:**
- Прогнать «Перестроить» на реальном корпусе (≥80 заметок). Через кнопку «Выгрузить» собрать markdown всех фасетов и просмотреть.
- Если модель всё ещё сыплет рассуждения → сменить `AI_FACET_MODEL` (env Cloud Functions) на `accounts/fireworks/models/gpt-oss-120b`, редеплой `summarizeFacet`, перепроверить. Сравнить качество qwen vs gpt-oss.
- Подобрать `DOMAIN_THRESHOLD` (сейчас 0.45): домены пустые → понизить (0.40); всё валится в один → поднять (0.50).
**Приёмка:** ≥80% карточек имеют связное ИИ-описание по своей теме; нет рассуждений/сырого текста; основные домены не пустые.

## PROF-2 — Наполнить домены-меньшинства (Дети/Родители/Коллеги)
**Проблема:** argmax-назначение чанка в ОДИН лучший домен + широкие сиды → редкие темы (про дочь, родителей, коллег) проигрывают «Самореализации»/«Деньгам» и пропадают.
**Варианты (реализовать 1-й, при нехватке — 2-й):**
1. **Мульти-назначение:** чанк попадает во ВСЕ домены, где cosine ≥ порога (а не только argmax). В `AIProfileFacetService.build` заменить argmax-блок на цикл по всем доменам с порогом. Риск: перекрытие — компенсировать чуть более высоким порогом для не-лучших доменов (напр. argmax всегда; плюс вторичные, если cosine ≥ `DOMAIN_THRESHOLD + 0.03`).
2. **Сузить/выправить сиды** в `lifeDomains.ts` (особенно «Самореализация» — она самая жадная) и поднять её порог индивидуально (сделать порог полем домена).
**Приёмка:** если в корпусе есть заметки про детей/родителей/коллег — соответствующие домены появляются с ≥2 заметками и осмысленным описанием.

## PROF-3 — Доказательность: открыть заметки темы
**Цель:** клик по фасету показывает его заметки-источники (у фасета уже есть `noteIds`).
**Действия:** в `ProfileFacets.tsx` при раскрытии карточки показать список заметок (title + дата) по `facet.noteIds` (читать `documents` из IndexedDB), каждая — ссылка/кнопка, открывающая заметку (как в `DocumentPickerModal`/архиве). Опц.: «найти в чате по этой теме» → прокинуть в поиск.
**Приёмка:** из карточки темы видно и можно открыть конкретные заметки.

## PROF-4 — Живой профиль (инкрементальное обновление)
**Цель:** профиль пополняется сам по мере письма, без полного пересчёта.
**Действия:**
- Хранить на фасете «грязный» флаг и центроид (центроид уже есть). При индексации новой заметки (`useEmbeddingIndexer`) — назначать её чанки к ближайшим фасетам (cosine к `centroid`), помечать затронутые фасеты грязными, добавлять `noteIds`.
- Фоновый проход: пересчитывать `summarizeFacet` ТОЛЬКО для грязных фасетов. Полную рекластеризацию (`build()`) запускать редко (раз в неделю / по кнопке).
- Тренды уже считаются по датам (`firstAt/lastAt` → бейдж новая/активна/затихла) — оставить.
**Приёмка:** новая заметка попадает в подходящие темы без ручного «Перестроить»; пересчитываются только изменённые темы.

## PROF-5 — Reduce-портрет (заменить «простыню»)
**Цель:** общий психологический портрет синтезируется ИЗ фасетов, а не из всех саммари сразу.
**Действия:** в `AIProfileService.generate()` (`src/features/ai/services/AIProfileService.ts`) вместо дампа всех `aiSummaries` собрать вход из фасетов (label + summary) и попросить LLM связный портрет. Лимит-экемпт путь (как `summarizeFacet`). Масштабируется на большой корпус.
**Приёмка:** портрет ссылается на реальные темы профиля, не раздувается на 500+ заметок.

## PROF-6 — Связка с чатом + кандидаты в посты
**Действия:**
- В `useAIChat.ts`: при релевантном запросе подмешивать в контекст не только найденные заметки, но и подходящие фасеты («в последнее время ты много пишешь про X»). Брать фасеты по близости запроса к `centroid`.
- **Кандидаты в посты:** на фасете считать «плотность инсайтов» (напр. доля заметок с непустыми `insights` в их `aiSummaries`, или отдельная оценка), показывать в UI «годится в пост» + кнопку «сделать пост» (исходный кейс пользователя).
**Приёмка:** чат опирается на темы; в профиле видно темы-кандидаты в посты.

## PROF-7 — Лица конкретных людей (entity-aware фасеты) — Фаза 2+
**Проблема:** конкретные люди (отец, мать, жена/Саша, дочь/Вика) упоминаются короткими кусками внутри дневника — кластеризация их не выделяет надёжно. **Пользователь явно хочет эту фичу.**
**Подход:** извлечение сущностей. Вариант: при суммаризации заметки (`summarizeDocument`) дополнительно вытаскивать упомянутых людей (имя + роль: жена/дочь/мать/коллега) → хранить per-note. Затем строить фасет на человека из заметок, где он упомянут, + краткое ИИ-описание «что пользователь пишет про <имя>». Нужен дедуп имён (Саша=жена, Вика=дочь и т.п.).
**Приёмка:** значимые люди появляются отдельными карточками с описанием отношений/паттернов.

## Порядок
PROF-1 (проверка/тюнинг) → PROF-2 (домены-меньшинства) → PROF-3 (доказательность) → PROF-4 (живость) → PROF-5 (портрет) → PROF-6 (чат/посты) → PROF-7 (лица).
PROF-1..3 быстрые и закрывают «чтобы хорошо выглядело». PROF-4..7 — развитие.

## Открытые решения
- `AI_FACET_MODEL`: qwen3-30b-a3b vs gpt-oss-120b — выбрать по PROF-1.
- Облачный синк фасетов: добавить, только если нужен мультидевайс (сейчас локально, регенерируемо).
- Пороги (`DOMAIN_THRESHOLD`, `MERGE_THRESHOLD`) калибруются на реальных данных — не зашивать «на глаз», проверять выгрузкой.
