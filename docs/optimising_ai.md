# Оптимизация ИИ-диалога — тикеты (для исполнителя / opencode)

Документ самодостаточный. Цель — ускорить ответ ИИ, сократить число сетевых/LLM-вызовов на сообщение, убрать антипаттерны RAG и дублирование. Без изменения продуктового поведения (поиск/фасеты/персоны работают как сейчас, просто быстрее и чище).

Проект: `justwriting` (React+TS+Vite; Firebase Functions; Vercel). Провайдер ИИ — Fireworks, активная чат-модель `deepseek-v4-flash` (reasoning-модель).

---

## 0. Контекст: как сейчас устроен один ответ

Оркестрация целиком в `src/features/ai/hooks/useAIChat.ts` → `sendMessage` (~720 строк). Этапы:
1. Лимиты (клиент `useAiLimitStore.remaining`; сервер `withinGlobalDailyLimit`+`checkDailyLimit`+`checkRateLimit`).
2. Оптимистичный показ сообщения.
3. Персона + портрет (портрет кэшируется в `portraitCacheRef`; кастом-персона — параллельно).
4. Детект намерения `looksLikeNoteSearch` (регексы) + «липкий» режим (`stickyTurnsRef`, 4 хода).
5. **Фасет-аугментация** (психо-персоны всегда, иначе при поиске): `AIService.embed(query)` → косинус к `facet.centroid` + boosts → топ-5 → блок «Темы профиля».
6. **Поиск** (при поиске): query expansion (LLM-вызов `coach`) → `searchNotesMulti(queries)` (гибрид вектор+BM25 RRF + cloud-рерэнк) → поиск по именам регексом по `chunkTexts` → объединение → рерэнк если >5 → бюджетирование (25K) → кап 30K.
7. Сборка system-промпта (сервер).
8. **Контекст вставляется как фейковый user-ход** `[Документ пользователя]…` + ответ-заглушка.
9. Генерация: `streamText` (`api/chat.ts`), фолбэк callable `chatWithAI.ts`, `maxOutputTokens 8192`.
10. Сохранение + авто-нейминг (ещё LLM-вызов на первом обмене).

**Ключевые файлы:** `src/features/ai/hooks/useAIChat.ts`, `src/features/ai/utils/noteRetriever.ts` (`searchNotes`, `searchNotesMulti` — сами эмбеддят запрос), `src/features/ai/services/AIService.ts` (`embed`/`chat`/`rerank`), `functions/src/ai/chatWithAI.ts` (callable), `api/chat.ts` (стрим), `functions/src/ai/summarizeFacet.ts` (эталон «дешёвая послушная модель» — `AI_FACET_MODEL`).

**Грабли (не наступать):**
- Инфра-вызовы (`embedDocument`, `rerankNotes`, `summarizeFacet`) — только `withinGlobalDailyLimit`, без `checkDailyLimit`/`checkRateLimit`.
- DeepSeek v4 — reasoning-модель, течёт «мышлением» в ответ; для служебных задач брать `AI_FACET_MODEL` (env), не активную чат-модель.
- Деплой функций: `cd functions && firebase deploy --only functions:<name>`; новая v2-callable → проверить `curl -X POST <url> -d '{"data":{}}'` = 401.
- Гейт перед коммитом: `npx tsc --noEmit` (root+`functions`), `npx eslint <файлы>`, `npm run build`. Не ломать слоевые границы.
- НЕ менять видимое поведение: те же результаты поиска/фасетов, просто быстрее.

## OPT-1 — Переиспользовать эмбеддинг запроса (сейчас считается 2–3×)
**Проблема:** на один поисковый вопрос запрос эмбеддится отдельно в фасет-аугментации (`useAIChat.ts` ~стр. 336, `AIService.embed`) и ещё раз внутри `searchNotesMulti` (`noteRetriever.ts` стр. 290). Плюс `AIEmbeddingService.getAll()` вызывается в нескольких местах за один ход.
**Задачи:**
- Добавить в `searchNotes`/`searchNotesMulti` опциональный параметр `queryVector?: number[]` (и `allEmbeddings?` опц.), использовать переданный вектор вместо повторного `embed`.
- В `sendMessage` посчитать эмбеддинг запроса ОДИН раз и прокинуть и в фасет-аугментацию, и в поиск.
- `AIEmbeddingService.getAll()` за один `sendMessage` вызывать один раз, результат переиспользовать (фасет-ноты, name-search, vector search).
**Приёмка:** на поисковый ход — максимум 1 `embed`-вызов запроса и 1 `getAll()`; результаты поиска не изменились.

## OPT-2 — Query expansion убрать с критического пути
**Проблема:** `useAIChat.ts` ~стр. 405–418 делает блокирующий LLM-вызов (`AIService.chat` персоной `coach` → reasoning-модель) ради 3 синонимов ДО поиска. Большая задержка, жжёт лимит, гибрид BM25+вектор и так даёт recall.
**Задачи (выбрать один путь, по умолчанию — A):**
- A. Сделать expansion **опциональным и выключенным по умолчанию** (флаг/эксперимент); поиск идёт по исходному запросу.
- B. Если оставлять — гонять **дешёвой моделью** (`AI_FACET_MODEL`) и **параллельно** с первичным вектор-поиском, а не строго до него; результаты досливать.
**Приёмка:** на типичный поиск нет лишнего блокирующего chat-вызова; время до начала ответа заметно меньше; recall не просел на проверочных запросах.

## OPT-3 — Авто-нейминг диалога дешёвой моделью
**Проблема:** `useAIChat.ts` ~стр. 644 после первого обмена зовёт `AIService.chat` персоной `coach` (reasoning-модель) для названия. Уже `void` (не блокирует), но это лишний дорогой вызов.
**Задачи:** перевести авто-нейминг на дешёвую модель. Либо новый лёгкий callable (как `summarizeFacet`, лимит-экемпт, `AI_FACET_MODEL`), либо параметр модели в существующем пути. Не блокировать UI.
**Приёмка:** название генерится дешёвой моделью, не трогает пользовательский чат-лимит, UI не ждёт.

## OPT-4 — Фасет-аугментация: не гонять на тривиальных сообщениях + кэш фасетов
**Проблема:** для психо-персон (`group_psychology`/`cbt`/`coach`) блок п.5 (`useAIChat.ts` ~стр. 325–398) исполняется на КАЖДОМ сообщении (даже «привет», «спасибо»): эмбед запроса + `AIProfileFacetService.getAll()` + чтение `documents`. Лишняя задержка в обычном разговоре.
**Задачи:**
- Гейтить аугментацию: пропускать очень короткие/несодержательные реплики (напр. < N слов и без вопросительных/референсных маркеров).
- Кэшировать `AIProfileFacetService.getAll()` и `documents` в ref на сессию диалога (инвалидировать при ребилде фасетов / смене диалога).
**Приёмка:** короткие реплики не запускают эмбед/чтение фасетов; фасеты читаются из БД один раз за сессию, а не каждый ход.

## OPT-5 — Контекст RAG в system-промпт, а не фейковым user-ходом
**Проблема:** и `chatWithAI.ts` (стр. 80–87), и `api/chat.ts` (стр. 302–309) вставляют `documentContent` как `unshift({role:'user'})` + ответ-заглушку «Документ получен». Модель может трактовать это как слова пользователя и отвечать «по документу», а не на вопрос; структура диалога засоряется.
**Задачи:**
- Передавать найденный контекст (фасеты+заметки) в **system-промпт** отдельной размеченной секцией (`[Контекст из заметок пользователя]\n…`), НЕ как user-ход. Реальный вопрос пользователя остаётся последним сообщением.
- Убедиться, что нет старого бага с обрезкой system на ~9.5K (если есть — поднять лимит; system-окно модели большое).
**Приёмка:** в `messages` к модели нет синтетического user-хода с документом; ответ по-прежнему опирается на контекст; нет обрезки.

## OPT-6 — Убрать дублирование двух бэкендов
**Проблема:** `api/chat.ts` (стрим) и `functions/src/ai/chatWithAI.ts` (callable) дублируют: сборку system-промпта (`buildSystemPrompt`/inline), лимиты, injection-гард, вставку `documentContent`, `responseLength`. Уже слегка разошлись → риск дрейфа.
**Задачи:** вынести общий конструктор запроса (system-промпт + подготовка `messages` + injection-гард) в `functions/src/shared` (или общий модуль, доступный обоим). Оба бэкенда вызывают его. Поведение не меняется.
**Приёмка:** одна реализация сборки промпта/гарда на оба пути; tsc обоих зелёный; ответы идентичны до/после.

## OPT-7 — Порядок и бюджет контекста («lost in the middle»)
**Проблема:** до 30K контекста кладётся ОДНИМ блоком в начало (через фейк-ход), вопрос — далеко в конце; самые релевантные заметки могут оказаться в «середине».
**Задачи:** в `useAIChat.ts` блоке сборки `noteBlock` — класть самые релевантные заметки (по composite score, уже считается ~стр. 506) ближе к концу контекста (рядом с вопросом). Перепроверить бюджет: при system-контексте (OPT-5) можно поднять кап.
**Приёмка:** топ-релевантная заметка не теряется в середине; на проверочных вопросах ответ точнее ссылается на неё.

## OPT-8 — Мелочи (необязательно)
- Семантический детект намерения поиска (косинус запроса к «поисковому» прототипу) в дополнение к регексам — ловить перефразы.
- Проверить, не течёт ли reasoning DeepSeek (`<think>`/«мы должны…») в сами ответы чата (как было в фасетах); при необходимости — чистка/реджект в `sanitizeAiResponse`.

## Порядок
OPT-1 (переиспользование эмбеддинга) → OPT-2 (expansion с критического пути) → OPT-4 (гейт/кэш фасетов) → OPT-3 (нейминг) → OPT-5 (контекст в system) → OPT-6 (дедуп бэкендов) → OPT-7 (порядок) → OPT-8.
OPT-1/2/4 дают основной выигрыш по скорости и лимитам. OPT-5/6 — чистота и поддерживаемость.

## Проверка эффекта
Логировать (console.time или временные таймстемпы) число `embed`/`chat`/`rerank`-вызовов и время до первого токена на типичном поисковом сообщении до и после. Цель: на поиск — 1 embed + (0–1) expansion + 1 rerank + 1 генерация; время до первого токена заметно меньше.
