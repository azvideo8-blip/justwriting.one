# Офлайн-устойчивость и синхронизация — тикеты для исполнителя (MiMo)

Самодостаточный файл. Префикс — `SYNC-`. Повод: 05.07.2026 квота записи Firestore привела к тому, что заметка не сохранялась и не переименовывалась, без единого сигнала пользователю о причине (см. `CHANGELOG.md` запись за 2026-07-05 — там уже чинили автосейв/эмбеддинги, это следующий слой проблемы: устойчивость записи метаданных + видимость статуса).

## Требование (от пользователя, дословно распакованное)
1. Приложение не должно падать/ломаться при проблемах с облачной БД (лимиты, недоступность, нет сети).
2. Всё, что опирается на локальную базу (написание, сохранение, редактирование, переименование, теги, даты) — должно работать полностью офлайн.
3. Если облако недоступно (сеть/лимиты/что угодно) — приложение продолжает работать; если автосинхронизация включена, где-то в приложении виден статус «синхронизация приостановлена»; правки копятся в единую очередь и уходят в облако сами, как только сеть/облако восстановятся.

## Контекст (уже есть в коде, проверено чтением)
- Архитектура local-first уже описана в `ARCHITECTURE.md` («Sync Flow»): запись всегда сначала в IndexedDB, потом (best-effort) в Firestore.
- `src/core/services/StorageService.ts` (`_doSaveVersion`) → локальная версия сохраняется первой; при неудаче локального сохранения — исключение (quota), при неудаче облака — **не бросает**, потому что `CloudSyncService.syncVersionToCloud` (`src/core/services/CloudSyncService.ts:275-363`) сам ловит любую ошибку и кладёт документ в очередь `syncQueue`. Этот путь (сохранение версий текста) уже устойчив — ломать не надо, но стоит закрепить тестом (SYNC-4).
- `src/core/firebase/firestore.ts` — `isFirestoreConnected` + `onConnectionChange`; `resource-exhausted` (квота) намеренно считается «сервер ответил» → флаг остаётся `true` (это ок для heartbeat, но означает, что квота НЕ трактуется как «офлайн» нигде ниже по стеку).
- `src/shared/hooks/useOnlineStatus.ts` — обёртка над `navigator.onLine`/`online`/`offline` событиями.
- Очередь: IndexedDB store `syncQueue` (`src/core/storage/localDb.ts`), дренаж — `SyncService._drainPendingQueue` (`src/core/services/SyncService.ts`). Автозапуск дренажа **только** из `src/features/writing/contexts/WritingSessionContext.tsx:116-120`, при переходе `isBrowserOnline` → true, и только пока открыта сессия письма. Ручной запуск — кнопка «Sync Now» в Настройках (`useSyncDiagnostics.tsx:270-282`, `SyncDiagnostics.tsx`).
- Тумблер автосинхронизации уже существует: `src/features/settings/components/AppTab.tsx:23-45`, ключ `STORAGE_KEYS.AUTO_SYNC_ENABLED` (`src/shared/constants/storageKeys.ts:21`), секция «Хранилище» (`settings_section_storage`). `SyncService.syncPending` уважает этот флаг (`SyncService.ts:38`).
- Диагностика синка (`useSyncDiagnostics.tsx`) уже различает статусы `synced/pending/mismatch/local_only/cloud_only/cloud_missing` и знает `queueCount` — хорошая база для индикатора, но сейчас видна только на отдельном экране настроек.

## Найденная реальная дыра (источник вчерашнего бага)
`src/features/archive/services/archiveCrud.ts` (`updateArchiveField`) — правка title/tags/date/labelId:
- Для документов **с локальной копией** (`_isLocal === true`, строки 20-47): облачная правка обёрнута в `.catch()`, который только репортит ошибку и один раз показывает toast `archive_cloud_sync_failed` (`useArchiveSessions.ts:91-93`). **В `syncQueue` ничего не кладётся** — при следующем открытии облако так и останется со старым названием/тегами, пока пользователь не поправит это же поле ещё раз.
- Для документов **без локальной копии** («Cloud Only», реальный статус в диагностике синка — не гипотетический, строки 50-64): вызов `DocumentService.updateTitle/...` идёт **без try/catch и без локального стора вообще**. Офлайн/квота → промис реджектится → `handleFieldUpdate` ловит исключение (`useArchiveSessions.ts:94-97`), локальный оптимистичный апдейт (`updateSession`) не происходит → переименование выглядит полностью проваленным. Это и есть «невозможно было переименовать».

## SYNC-1 — Офлайн-фолбэк для правок «Cloud Only» заметок 🔴
**Задачи:**
1. В `updateArchiveField` для ветки без локальной копии (`_isLocal === false`): при первой правке любого поля (title/tags/date/labelId) прозрачно создавать локальную копию документа в IndexedDB (переиспользовать логику из `CloudSyncService.addLocalCopy` — подтянуть метаданные + версии, проставить `linkedCloudId`), затем применять правку через уже существующий локальный путь (`LocalDocumentService.updateTitle/updateTags/updateDate/updateLabelId`), а в облако — best-effort, как у обычных линкованных документов.
2. Если создание локальной копии само не удалось (нет сети вообще, не квота) — не терять правку молча: минимум не показывать общий "провал", а fallback-сообщение в духе текущего `archive_cloud_sync_failed`.
3. Для уже существующей ветки с локальной копией (строки 20-47): при ошибке облачного вызова — вместо только toast, класть задачу в `syncQueue` (тип `document`, тот же формат, что использует `SyncService.addToQueue`/`syncVersionToCloud`) на этот `documentId`, чтобы дренаж (SYNC-2) реально дотащил правку до облака, а не терял её после разового тоста.
**Приёмка:** без сети или при `resource-exhausted` от Firestore переименование/теги/дата/метка у ЛЮБОЙ заметки (включая «Cloud Only») применяются мгновенно, без ошибки в UI; после восстановления связи изменение само уезжает в облако без повторного действия пользователя.

## SYNC-2 — Глобальный автодренаж очереди синхронизации 🟠
**Диагноз:** сейчас дренаж стартует только из `WritingSessionContext.tsx:116-120` (переход browser offline→online, и только при открытой сессии письма). Если пользователь сидит в архиве/настройках — не сработает. Если `navigator.onLine` не менялся весь день (сеть была), а Firestore квота освободилась (`isFirestoreConnected` восстановился) — тоже не сработает, потому что триггер завязан на браузерное online-событие, а не на восстановление самого Firestore.
**Задачи:**
1. Поднять подписку на `onConnectionChange` (`core/firebase/firestore.ts`) и на `useOnlineStatus` на уровень приложения (например, в `AppProviders.tsx` — отдельный лёгкий sync-manager-провайдер), не привязываясь к монтированию экрана письма.
2. При переходе `isFirestoreConnected` false→true ИЛИ browser offline→online (и пользователь не гость, и `auto_sync_enabled !== 'false'`) — вызывать `SyncService.syncPending(userId)`. Гонки не страшны — в `SyncService` уже есть `_syncInProgress`-гвард.
3. Существующий вызов в `WritingSessionContext.tsx` можно оставить (безопасен благодаря гварду) или перенести в общий провайдер — на усмотрение исполнителя, лишь бы не плодить независимые таймеры.
**Приёмка:** очередь синхронизации автоматически дренируется при восстановлении сети или самого Firestore, независимо от того, какой экран открыт, без ручного «Sync Now» в Настройках.

## SYNC-3 — Индикатор «Синхронизация приостановлена» 🟠
**Задачи:**
1. Общий хук (`useSyncStatus`) на базе: `auto_sync_enabled`, `isFirestoreConnected`/`onConnectionChange`, `useOnlineStatus`, `SyncService.getPendingCount()`. Статус «приостановлена»: `auto_sync_enabled !== 'false'` И (офлайн ИЛИ `!isFirestoreConnected` ИЛИ `pendingCount > 0` дольше ~30 с). Если автосинк выключен пользователем — индикатор вообще не показывать.
2. Глобальное лёгкое место — иконка/бейдж в общем layout, видимый с любого экрана (посмотреть готовые точки в `src/features/navigation`, шапка или нижняя навигация). Тап/клик — ведёт в Настройки → Хранилище, к блоку автосинка.
3. В самих Настройках (`AppTab.tsx`, рядом с `settings_auto_sync`) — текстовая строка статуса: синхронизировано / приостановлена (нет сети) / приостановлена (облако недоступно) / «N заметок ждут отправки». Новые i18n-ключи — в `src/shared/i18n/translations/settings.ts`, пара ru+en, по образцу соседних.
4. Без спама toast на каждую фоновую ошибку — только устойчивый статус-индикатор, который сам исчезает, когда очередь пуста и связь есть.
**Приёмка:** при включённой автосинхронизации и недоступной сети/облаке индикатор виден с любого экрана; при восстановлении и пустой очереди — пропадает сам, без участия пользователя.

## SYNC-4 — Тесты, закрепляющие «не падает» 🟡
**Задачи:**
1. Тест: `StorageService.saveVersion` продолжает успешно сохранять локально и не бросает исключение, если облачный вызов реджектится с `resource-exhausted` и отдельно с `unavailable`/офлайн (это уже так работает — тест фиксирует поведение, чтобы SYNC-1..3 его не сломали).
2. Тест: `updateArchiveField` после фикса SYNC-1 — при недоступном облаке (оба случая: с локальной копией и «Cloud Only») правка не бросает исключение, применяется локально, факт неудачи облака корректно долетает до статуса синка (SYNC-3), ничего не всплывает до `ErrorBoundary`.
3. Быстрая ревизия остальных мест прямого чтения из Firestore на основных экранах письма/сохранения (не архив — там уже есть `cloudLoadFailed`, см. `useArchiveSessions.ts:31`) — на предмет голого throw без fallback на локальные данные.
**Приёмка:** прогон тестов с эмулируемым офлайном/`resource-exhausted` подтверждает, что письмо, сохранение, редактирование метаданных и просмотр заметок не роняют экран в `ErrorBoundary` и не подвисают.
