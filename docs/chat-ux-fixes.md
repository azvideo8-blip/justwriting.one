## UXFIX-1 — «Смена объёма» не меняет ответ 🔴
**Диагноз (проверено):** длина не применяется, потому что её перебивает **жёсткий формат самой персоны**. В каждом персона-промпте (`prompts.ts`) зашито:
```
RESPONSE FORMAT:
- Keep responses concise: 150–250 words maximum for chat replies.
```
плюс фиксированная многосекционная структура (особенно `group_psychology` — панель из 4 терапевтов + 4 секции). Инструкция длины добавляется в `buildChatSystemPrompt` мягким суффиксом («ВАЖНО: кратко/подробно…») ПОСЛЕ персоны, и конфликтует с её же «150–250 max» → модель выдаёт примерно одинаковый персона-ответ при любом объёме (а на деле и кэп игнорит — ответы длиннее 250).
Персист объёма работает (`updateResponseLength` пишет `dialogue.responseLength`, `useAIChat:741` берёт `dialogue?.responseLength || responseLength`), так что причина — приоритет формата персоны, не стейт.
**Задачи:**
1. Убрать/смягчить жёсткий «150–250 words maximum» из персона-промптов (в ОБОИХ файлах) — длину должен задавать слой `responseLength`, а не персона. Оставить в персоне только тон/роль/структуру-направление, без фиксированного словесного кэпа.
2. В `buildChatSystemPrompt` сделать директиву длины **авторитетной и последней**, с явным приоритетом: short = 1–2 абзаца, без секций; standard = умеренно; detailed = развёрнуто; reasoning = видимый `<reasoning>` + `<answer>`. Формулировать сильно («это указание о длине имеет приоритет над форматом выше»).
3. Для `group_psychology` сделать структуру **зависимой от объёма**: short → один синтезированный голос (не вся панель); detailed/reasoning → полная панель. Иначе короткий объём невозможен.
4. Проверить, что `reasoning` у `group_psychology` реально даёт отдельный сворачиваемый «ход мысли», а не панель внутри ответа (согласовать встроенные в персону теги `<reasoning>/<answer>` с парсером режима).
**Приёмка:** переключение Кратко/Стандартно/Объёмно/С рассуждением заметно и предсказуемо меняет длину и форму ответа у всех персон, включая «Группу психологов».

## UXFIX-2 — Индикатор «думает…» выглядит как зависание 🟠
**Проблема:** во время ожидания показывается статичный текст «Группа психологов думает…» — непонятно, жив ли процесс (скрин). 
**Задачи:** в состоянии загрузки (когда `isLoading` и ещё нет `streamingMessage`) показать **живую анимацию** (пульсирующие точки/shimmer) + **счётчик секунд** ожидания («думает… 7с»). Когда пошёл стрим (`streamingMessage`/`streamingReasoning`) — счётчик убрать, показывать сам поток. Не блокировать ввод.
**Приёмка:** во время ожидания что-то анимируется и идёт секундомер; очевидно, что приложение не зависло.

## UXFIX-3 — Массовый анализ всех заметок ИИ 🟠
**Проблема:** много заметок не проанализировано ИИ (нет саммари); сейчас только по-одной кнопкой «Analyze» (скрин). Нужна кнопка «Проанализировать все» и/или фоновый прогон.
**Задачи:**
1. Кнопка **«Проанализировать все»** (в архиве/диагностике): находит заметки без саммари (`AISummaryService.hasAll` / отсутствие записи), по очереди вызывает `AIService.summarize` → `AISummaryService.save` (как `handleAIClick`), с **прогрессом** и бэкоффом на лимитах. После — обновить карту `aiProcessedMap`.
2. **Важно про лимиты:** `summarizeDocument` сейчас вешает `checkDailyLimit` (5/сутки) + `checkRateLimit` (кулдаун 10с) → массовый прогон 80 заметок упрётся в 429. Варианты: (а) вывести `summarizeDocument` из-под пер-юзер лимитов, оставив только `withinGlobalDailyLimit` (как `embedDocument`/`rerankNotes`/`summarizeFacet` — это фоновый анализ, не чат) — **рекомендую**; (б) если оставлять лимит — прогон с уважением кулдауна (медленно) + бэкофф. См. эталон лимит-экемпт в `functions/src/ai/embedDocument.ts`.
3. Опц.: автозапуск фонового добора саммари при простое (как индексатор эмбеддингов `useEmbeddingIndexer`), по 1–3 за тик, бэкофф.
**Приёмка:** одной кнопкой можно проанализировать все непроанализированные заметки с прогрессом; массовый прогон не утыкается в 429; (опц.) фон сам добирает.
